# KYC Network Scout

Автоматизований OSINT-пайплайн для KYC / Due Diligence перевірки українських
компаній та фізичних осіб. Система збирає дані з незалежних джерел, оцінює
ризик контрагента і генерує структурований звіт зі збереженням усіх первинних
доказів (evidence). Працює як через **веб-інтерфейс** (форма в браузері з
пакетною перевіркою кількох контрагентів), так і через **CLI**.

> ⚠️ **Важливо.** Звіт системи — це **допоміжний інструмент аналітика**, а не
> автоматичний вердикт. Остаточне KYC-рішення приймає людина на основі цих
> даних та власного професійного судження. Система не призначена для повністю
> автоматизованого прийняття рішень про відмову клієнту.

---

## 1. Опис

KYC Network Scout перевіряє суб'єкта (компанію або фізособу) за трьома
незалежними напрямами і зводить результат у єдиний звіт:

| Worker | Джерело | Що перевіряє |
|--------|---------|--------------|
| **Worker 1 — Реєстр** | Opendatabot (держреєстри України) | реєстраційні дані, бенефіціари, керівник, ризик-фактори (RF/BY/IR власники, санкції, банкрутство, бойова територія, PEP, регуляторні стягнення) |
| **Worker 2 — Санкції** | Opendatabot (`/person-sanctions` + реєстрові factors) | збіги у санкційних списках: РНБО, США (SDN/Non-SDN/OFAC), ЄС, Велика Британія, Канада |
| **Worker 3 — Adverse media** | Serper + Claude | негативні згадки у відкритих медіа (пошук за брендом/торговою маркою — укр. і англ., а не за юр-назвою) |

На виході — інтегральна **оцінка ризику (0–100, де 0 = мінімальний ризик,
100 = максимальний)**, рівень ризику (LOW / MEDIUM / HIGH / INCONCLUSIVE),
список red flags, **audit trail** (покроковий розклад, як накопичувався ризик),
граф зв'язків та evidence-архів.

### Ключові особливості

- **Чесність про власні межі (INCONCLUSIVE).** Рівень «низький ризик»
  присвоюється лише якщо перевірено **і реєстр, і санкції**. Неповна перевірка
  не дає підстав назвати контрагента надійним — система каже INCONCLUSIVE
  замість фальшивого LOW (і не показує заспокійливе низьке число ризику).
- **Відсів однофамільців.** Негативна медіа-згадка враховується у скорингу
  лише якщо класифікатор підтвердив, що вона саме про цього суб'єкта
  (`is_about_target_person`) з достатньою впевненістю. Стаття про тезку не
  знижує бал — лише показується з позначкою «не враховано».
- **Чесне покриття джерел.** Звіт показує, що саме перевірено, а що ні.
  «Не знайдено» ≠ «чисто»; «джерело недоступне» ≠ «суб'єкта не існує».
- **List-complete санкційний скринінг (фізособи).** Кожен критичний санкційний
  список (РНБО, OFAC/SDN, ЄС, UK, Канада) запитується **окремо**, щоб жоден не
  випав за межі вікна результатів. Для компаній перевіряються санкційні фактори
  українського реєстру (мультиюрисдикційний скринінг — лише для фізосіб; звіт
  чесно це зазначає).
- **Медіа-пошук за брендом, а не юр-назвою.** Adverse-media гілка чистить
  орг-форму й лапки та шукає за брендом — українським **і** англійським
  написанням. Це усуває критичний false negative, коли пошук за юр-назвою не
  знаходить нічого, хоча компанія фігурує в розслідуваннях.
- **Санкція ≠ регуляторне стягнення.** Штрафи НБУ/АМКУ (`nbuSanctions`)
  трактуються як регуляторна історія, а **не** як санкційний збіг. Контрагент
  із адмін-штрафами не позначається хибно як «під санкціями».
- **Audit trail.** Кожен бал можна «розібрати назад» — крок за кроком, із
  зазначенням джерела кожного фактора (для відтворюваності рішення).
- **Пакетна перевірка.** Через веб можна за один раз перевірити до 5
  контрагентів — перевірки виконуються паралельно, звіти з'являються в міру
  готовності.
- **RF/BY/IR фокус.** Пріоритетне виявлення зв'язків з державами-агресорами.

### Чому код, а не лише ШІ-асистент

ШІ-асистент (Claude, GPT тощо) гнучкий, але **недетермінований** — він щоразу
перевіряє трохи інакше, по-різному інтерпретує дані і може пропустити окремий
санкційний список, бо «дивиться» на верхні результати видачі. Для compliance
це неприйнятно: перевірка має бути **відтворюваною** — та сама вхідна точка дає
той самий результат, із зафіксованими доказами.

KYC Network Scout це гарантує:
- **Детермінованість** — завжди ті самі джерела, той самий перебір усіх
  критичних санкційних списків, та сама формула скорингу. Якщо регулятор
  спитає «як саме ви перевіряли?», відповідь — код, а не «асистент подивився».
- **Гарантована повнота** — список-комплітний санкційний скринінг не «забуває»
  і не «лінується»: кожен список запитується окремо.
- **Докази** — `evidence.zip` фіксує всі первинні джерела на момент перевірки.
- **Швидкість і вартість на масштабі** — детерміновані прогони замість дорогих
  ШІ-міркувань над кожним HTTP-запитом.

Це **не заміна** ШІ-асистента, а доповнення. Найсильніша архітектура — ШІ як
оркестратор, що делегує Scout механічну, масову, детерміновану роботу, а сам
займається інтерпретацією контексту, нестандартними випадками й фінальним
висновком.

## 2. Напрям і задача

**Напрям 2 — Due Diligence: компанії та промислові об'єкти.**

**Задача:** інвестор або compliance-офіцер перевіряє контрагента перед
підписанням договору — треба швидко зрозуміти, чи є компанія реальним гравцем,
хто за нею стоїть і чи немає публічних сигналів ризику. Дано мінімальний вхід
(код, ПІБ або назва) — система автоматично збирає з офіційних та відкритих
джерел інформацію, оцінює ризик і зберігає докази у відтворюваному форматі.

## 3. Як встановити

### Передумови
- Docker і Docker Compose
- API-ключі: `OPENDATABOT_API_KEY`, `SERPER_API_KEY`, `ANTHROPIC_API_KEY` (розділ 7)

### Кроки

```bash
git clone https://github.com/LAKTIKO/kyc-network-scout.git
cd kyc-network-scout

# створити .env з ключами
cp .env.example .env
# відредагувати .env: вписати 3 ключі; для команди — задати KYC_WEB_PASSWORD

# зібрати образ
docker compose build
```

> Збірка тягне важкі залежності (Chromium для скрейпінгу) — перший build може
> зайняти 10–15 хвилин залежно від швидкості з'єднання.

## 4. Як запустити

### Варіант A — веб-інтерфейс (рекомендовано)

```bash
docker compose up -d        # піднімає web (:8000) + redis + воркери
# відкрити http://localhost:8000
```

У браузері: вкладки **Компанія** / **Фізособа**, форма з полями (юр-назва +
ЄДРПОУ або ПІБ + РНОКПП), кнопка **«+ Додати контрагента»** для пакетної
перевірки (до 5 за раз). Результати з'являються списком, кожен — з рівнем
ризику й посиланням на повний звіт.

**Розгортання для команди:** підняти інстанс один раз на внутрішній машині,
задати `KYC_WEB_PASSWORD` у `.env` і дати колегам адресу + пароль — їм нічого
встановлювати, лише браузер. Без пароля доступ відкритий (зручно локально, але
не для відкритого інтернету).

### Варіант B — CLI

Вхід визначається автоматично: 8 цифр → ЄДРПОУ (компанія), 10 цифр → РНОКПП
(фізособа), текст → ім'я особи (`--person`) або назва компанії (резолвиться).

```bash
docker compose run --rm kyc 14360570                                    # компанія за ЄДРПОУ
docker compose run --rm kyc "ПриватБанк"                                # компанія за назвою
docker compose run --rm kyc "Дерипаска Олег" --person                   # особа за іменем
docker compose run --rm kyc 1234567890 --person --name "Шевченко О. В." # особа за РНОКПП
docker compose run --rm kyc 14360570 --no-media                         # без adverse-media (швидше)
docker compose run --rm kyc 14360570 -v                                 # детальний лог
```

CLI-оркестратор (`run_kyc`) працює синхронно і не потребує Redis — найпростіший
шлях для одиничних перевірок. Celery-воркери з Redis передбачені для
масштабованого асинхронного режиму.

### Артефакти звіту

Веб віддає звіт прямо в браузері; CLI зберігає його в `./output/<timestamp>/`:
- `kyc_report.json` — повний структурований результат
- `report.html` — людиночитний звіт (адаптивний дизайн)
- `graph.html` — інтерактивний граф зв'язків (самодостатній)
- `evidence.zip` — усі первинні дані (raw + normalized)

| Вхід | Інтерпретація | Результат |
|------|---------------|-----------|
| `14360570` | ЄДРПОУ (8 цифр) | перевірка компанії |
| `"Дерипаска Олег" --person` | ім'я особи | санкційний + медіа скринінг |

| `"ПриватБанк"` | назва компанії | резолвинг → кандидати або перевірка |

Резолвинг назви чистить організаційні форми (ТОВ/LLC/JSC), лапки, пробує
варіанти транслітерації. При неоднозначності повертає кандидатів і просить
уточнити ЄДРПОУ (не вгадує — хибна ідентифікація небезпечніша за її відсутність).

## 6. Приклад вихідного звіту

Приклад знеособленого звіту наведено в `examples/` (HTML, граф, JSON).
Презентаційний лендинг із вбудованим прикладом — `landing/index.html`.

Консольний підсумок CLI-прогону для санкційної особи:

```
============================================================
  KYC ЗВІТ: Дерипаска Олег
============================================================
  Тип:          person
  Оцінка ризику: 68/100  (0 = мін, 100 = макс)
  Рівень:       HIGH
  Покриття:     sanctions=checked, adverse_media=checked
  Артефакти:    json, html, graph, evidence_zip
============================================================
```

Звіт містить: оцінку ризику + рівень із причиною; покриття джерел; red flags
(згруповані за вагою: критично / застереження / контекст); audit trail; дані
суб'єкта; санкційний скринінг (видно перевірені списки + джерело); регуляторну
історію; корпоративну структуру з графом; негативні медіа з посиланнями на
першоджерела; секцію «Відомі обмеження».

## 7. Джерела та API

| Сервіс | Призначення | Ключ (.env) |
|--------|-------------|-------------|
| **Opendatabot** | держреєстри України (компанії, ФОП, санкції РНБО/OFAC/ЄС/UK/Канада, PEP, регуляторні стягнення) | `OPENDATABOT_API_KEY` |
| **Serper** | пошук adverse media (Google SERP) | `SERPER_API_KEY` |
| **Anthropic Claude** | класифікація медіа-згадок | `ANTHROPIC_API_KEY` |

Санкційний скринінг повністю через Opendatabot — окремий ключ OpenSanctions
**не потрібен**. Ендпоінт `/person-sanctions` повертає міжнародні списки
(РНБО, США SDN/Non-SDN, ЄС, Велика Британія, Канада).

Веб-сервіс додатково читає `KYC_WEB_USER` / `KYC_WEB_PASSWORD` (авторизація)
та необов'язкові `KYC_MAX_INFLIGHT` / `KYC_MAX_JOBS` (ліміти).

**Свіжість даних:** Opendatabot проксіює офіційні держреєстри України — дані
актуальні на момент запиту.

## 8. Безпека та обмеження ресурсів

- **Авторизація.** Веб закривається паролем через `KYC_WEB_PASSWORD` (basic-auth
  на весь застосунок, крім healthcheck). Без пароля — відкрито; для деплою
  команді пароль обов'язковий, і порт не слід публікувати у відкритий інтернет.
- **SSRF-захист.** URL із пошукової видачі перед скрейпінгом перевіряються:
  лише http(s) на публічні IP — внутрішні/loopback/метадані хмари блокуються.
- **Ліміти.** Кап одночасних перевірок (захист API-кредитів) і кап
  in-memory стору задач; завершені звіти зберігаються на диск і переживають
  рестарт сервера.

## 9. Відомі обмеження

Система свідомо проектувалась із чесним відображенням своїх меж:

- **Закриті реєстри.** Оборонні / військові підприємства та об'єкти критичної
  інфраструктури законно приховані з відкритих держреєстрів — повну реєстрову
  картину по них отримати неможливо; це позначається, а не видається за «чисто».
- **UA-центричність.** Покриття обмежене українськими реєстрами. Іноземний
  суб'єкт у реєстрі **не знайдеться** — і це «не знайдено», а **не** «чисто».
- **Один реєстр — одна точка відмови.** Реєстрова й санкційна частини тримаються
  на Opendatabot; перехресної звірки за другим незалежним джерелом немає.
- **Санкції компаній — поверхневіші за фізосіб.** Для компанії перевіряється
  лише реєстровий sanction-фактор, без окремого мультиюрисдикційного скринінгу
  (OFAC/ЄС/UK/Канада) — звіт це чесно зазначає.
- **Судова статистика ≠ ризик.** Велика кількість судових справ у великих
  організацій (банки, ритейл) — наслідок масштабу. Релевантність справ (роль
  сторони, предмет, масштаб) поки не класифікується — у дорожній карті.
- **Тезки без ідентифікатора.** Перевірка особи за самим лише ПІБ ризикує хибною
  атрибуцією. Медіа-класифікатор відсіює однофамільців; INCONCLUSIVE не дає
  «чисто» на неоднозначних даних.
- **Adverse media.** Якість залежить від пошукової видачі та класифікатора;
  можливі хибнопозитивні та пропущені згадки.
- **Межа автоматизації.** Система — помічник, не заміна аналітика; фінальне
  рішення за людиною (звідси INCONCLUSIVE і явний дисклеймер у звіті).

## Архітектура

```
            ┌─ веб (FastAPI, :8000) ── форма + пакетна перевірка ─┐
вхід ───────┤                                                     ├─→ run_kyc
(ЄДРПОУ /   └─ CLI (docker compose run kyc …) ───────────────────┘      │
 РНОКПП /                                                                │
 ім'я /      ┌───────────────────────────────────────────────────────────┘
 назва)      │
             ├─ input_resolver ── тип + резолвинг назви → ЄДРПОУ (асиметрично)
             ├─ Worker 1 (registry_task / person_task) ── Opendatabot (реєстр)
             ├─ Worker 2 (sanctions_task) ────────────── Opendatabot (санкції)
             ├─ Worker 3 (run_adverse_media) ─────────── Serper + Claude (медіа)
             │
             └─ aggregator ── entity resolution + скоринг ризику + audit trail
                              → JSON + HTML + граф + evidence ZIP
```

Усі джерела зшиваються під єдиним `slug` (напр. `company_14360570`,
`person_<рнокпп>`). Оркестратор `run_kyc` працює синхронно (реєстр → санкції →
медіа → звіт). Веб-сервіс запускає `run_kyc` у фоновому пулі й віддає готовий
звіт; завершені задачі персистяться, тож посилання переживають рестарт.

**Основні модулі:** `workers/` (пайплайн і скоринг), `webapp/` (веб-міст),
`landing/` (лендинг + білдер самодостатнього HTML), `examples/` (приклад звіту).

### Бачення та дорожня карта

Повна стратегічна карта — у [`docs/ROADMAP.md`](docs/ROADMAP.md).

**Куди рухаємось:** не агрегатор реєстрів, а **внутрішня платформа-розслідування**
(entity-intelligence engine) — «внутрішній Палантір» команди. Центральний об'єкт —
**сутність та її граф зв'язків** (а не звіт); розслідування **pivot-оване** (від
вузла до вузла); скоринг ризику — одна **лінза**, не продукт. Рів — наскрізна
**трасабельність** і чесне розрізнення **доказ / сигнал / лід** (tier'и довіри
джерел), що дозволяє OSINT-ширину без втрати захищеності висновку.

Фази (цеглини платформи): 1) контракт Scout↔скіл + провенанс; 2) граф зв'язків
кола-1; 3) іноземна юрисдикція + OSINT-розширення джерел; 4) спільна БД як
object-store сутностей + версіонування (піврічні переперевірки з diff).
