# KYC Network Scout

Автоматизований OSINT-пайплайн для KYC / Due Diligence перевірки українських
компаній та фізичних осіб. Система збирає дані з незалежних джерел, оцінює
ризик контрагента і генерує структурований звіт зі збереженням усіх первинних
доказів (evidence).

> ⚠️ **Важливо.** Звіт системи — це **допоміжний інструмент аналітика**, а не
> автоматичний вердикт. Остаточне KYC-рішення приймає людина на основі цих
> даних та власного професійного судження. Система не призначена для повністю
> автоматизованого прийняття рішень про відмову клієнту.

---

## 1. Опис

KYC Network Scout перевіряє суб'єкта (компанію або фізособу) за трьома
незалежними напрямами і зводить результат у єдиний звіт:

| Worker | Джерело | Що перевіряє |
|--------|---------|--------------|
| **Worker 1 — Реєстр** | Opendatabot (держреєстри України) | реєстраційні дані, бенефіціари, керівник, ризик-фактори (RF/BY/IR власники, санкції, банкрутство, бойова територія, PEP, регуляторні стягнення) |
| **Worker 2 — Санкції** | Opendatabot (`/person-sanctions` + реєстрові factors) | збіги у санкційних списках: РНБО, США (SDN/Non-SDN/OFAC), ЄС, Велика Британія, Канада |
| **Worker 3 — Adverse media** | Serper + Claude | негативні згадки у відкритих медіа (пошук за брендом/торговою маркою — укр. і англ., а не за юр-назвою) |

На виході — інтегральний **trust score (0–100)**, рівень ризику
(LOW / MEDIUM / HIGH / INCONCLUSIVE), список red flags, **audit trail**
(покроковий розклад, як сформувався бал), граф зв'язків та evidence-архів.

### Ключові особливості

- **Чесність про власні межі (INCONCLUSIVE).** Рівень «низький ризик»
  присвоюється лише якщо перевірено **і реєстр, і санкції**. Неповна перевірка
  не дає підстав назвати контрагента надійним — система каже INCONCLUSIVE
  замість фальшивого LOW. Це головна особливість: інструмент відмовляється
  видавати висновок, під яким аналітик не зможе підписатися.
- **Чесне покриття джерел.** Звіт показує, що саме перевірено, а що ні.
  «Не знайдено» ≠ «чисто»; «джерело недоступне» ≠ «суб'єкта не існує» —
  ці стани розрізняються явно.
- **List-complete санкційний скринінг.** Кожен критичний санкційний список
  (РНБО, OFAC/SDN, ЄС, UK, Канада) запитується **окремо**, щоб жоден не випав
  за межі вікна результатів. Це усуває тихий false negative, коли OFAC/SDN
  губиться серед численних РНБО-записів.
- **Медіа-пошук за брендом, а не юр-назвою.** Преса пише про компанію за
  торговою маркою, а не за повною юридичною назвою з орг-формою
  («ТОВАРИСТВО З ОБМЕЖЕНОЮ ВІДПОВІДАЛЬНІСТЮ "…"»). Тому adverse-media гілка
  чистить орг-форму й лапки та шукає за брендом — українським **і**
  англійським написанням. Це суттєво підвищує простежуваність згадок у медіа
  й усуває критичний false negative, коли пошук за юр-назвою не знаходить
  нічого, хоча компанія фігурує в розслідуваннях.
- **Санкція ≠ регуляторне стягнення.** Штрафи НБУ/АМКУ (`nbuSanctions`)
  трактуються як регуляторна історія (зважена за свіжістю/типом/тематикою),
  а **не** як санкційний збіг. Контрагент із адмін-штрафами не позначається
  хибно як «під санкціями».
- **Асиметричний принцип резолюції.** При неоднозначному вході (назва, що
  відповідає кільком компаніям) система **не вгадує**, а повертає список
  кандидатів для уточнення. Хибна ідентифікація небезпечніша за її відсутність.
- **Audit trail.** Кожен бал можна «розібрати назад» — крок за кроком, із
  зазначенням джерела кожного фактора. Важливо для відтворюваності рішення.
- **RF/BY/IR фокус.** Пріоритетне виявлення зв'язків з державами-агресорами.

### Чому код, а не лише ШІ-асистент

ШІ-асистент (Claude, GPT тощо) гнучкий, але **недетермінований** — він щоразу
перевіряє трохи інакше, по-різному інтерпретує дані і може пропустити окремий
санкційний список, бо «дивиться» на верхні результати видачі. Для compliance
це неприйнятно: перевірка має бути **відтворюваною** — та сама вхідна точка дає
той самий результат, із зафіксованими доказами.

KYC Network Scout це гарантує:
- **Детермінованість** — завжди ті самі джерела, той самий перебір усіх
  критичних санкційних списків, та сама формула скорингу. Якщо регулятор
  спитає «як саме ви перевіряли?», відповідь — код, а не «асистент подивився».
- **Гарантована повнота** — список-комплітний санкційний скринінг не «забуває»
  і не «лінується»: кожен список (РНБО, OFAC/SDN, ЄС, UK, Канада) запитується
  окремо. Та сама пастка, де OFAC тихо губиться у видачі, для ШІ-пошуку
  реальна, а для Scout — закрита кодом.
- **Докази** — `evidence.zip` фіксує всі первинні джерела на момент перевірки.
  Якщо рішення оскаржать через рік, є архів «ось що ми бачили тоді».
- **Швидкість і вартість на масштабі** — перевірка контрагента з колом
  пов'язаних осіб через міркування ШІ коштує багато токенів і часу; через код —
  це швидкі детерміновані прогони.

Це **не заміна** ШІ-асистента, а доповнення. Найсильніша архітектура — ШІ як
оркестратор, що делегує Scout механічну, масову, детерміновану роботу (зібрати
реєстр, перебрати санкції, порахувати скоринг, зберегти докази), а сам
займається тим, що код не вміє: інтерпретацією контексту, нестандартними
випадками, іноземними юрисдикціями, фінальним висновком. Scout звільняє ШІ від
потреби «думати» над кожним HTTP-запитом — це швидше, дешевше і надійніше.

## 2. Напрям і задача

**Напрям 2 — Due Diligence: компанії та промислові об'єкти.**

**Задача:** інвестор або compliance-офіцер перевіряє контрагента перед
підписанням договору — треба швидко зрозуміти, чи є компанія реальним гравцем,
хто за нею стоїть і чи немає публічних сигналів ризику. Дано мінімальний вхід
(код, ПІБ або назва) — система автоматично збирає з офіційних та відкритих
джерел інформацію, релевантну для KYC/AML, оцінює ризик і зберігає докази у
відтворюваному форматі. Ручна перевірка контрагента за десятком джерел займає
години; пайплайн робить базову частину за хвилини, лишаючи аналітику фінальне
судження.

### Передумови
- Docker і Docker Compose
- API-ключі (див. розділ 7)

```bash
git clone https://github.com/LAKTIKO/kyc-network-scout.git
cd kyc-network-scout

# створити .env з ключами
```

## 4. Як запустити

Система запускається однією командою через сервіс `kyc`. Вхід визначається
автоматично: 8 цифр → ЄДРПОУ (компанія), 10 цифр → РНОКПП (фізособа), текст →
ім'я особи (`--person`) або назва компанії (резолвиться в ЄДРПОУ).

```bash
# компанія за ЄДРПОУ
docker compose run --rm kyc 14360570

# фізособа за іменем
docker compose run --rm kyc "Дерипаска Олег" --person

# фізособа за РНОКПП
docker compose run --rm kyc 1234567890 --person --name "Шевченко Олександр"

# компанія за назвою (резолвинг → ЄДРПОУ)
docker compose run --rm kyc "ПриватБанк"

# без adverse-media гілки (швидше, менше API)
docker compose run --rm kyc 14360570 --no-media

# детальний лог
docker compose run --rm kyc 14360570 -v
```

Звіт зберігається в `./output/<timestamp>/`:
- `kyc_report.json` — повний структурований результат
- `report.html` — людиночитний звіт
- `report.pdf` — те саме у PDF
- `graph.html` — інтерактивний граф зв'язків
- `evidence.zip` — усі первинні дані (raw + normalized)

### Production-режим (опційно)

Для масштабованого асинхронного оброблення передбачено Celery-воркери з Redis:

```bash
docker compose up -d   # піднімає redis + воркери + scheduler
```

CLI-оркестратор (`run_kyc`) працює синхронно і не потребує Redis — це
найпростіший шлях для одиничних перевірок.

## 5. Приклад вхідних даних

| Вхід | Інтерпретація | Результат |
|------|---------------|-----------|
| `14360570` | ЄДРПОУ (8 цифр) | перевірка компанії |
| `"Дерипаска Олег" --person` | ім'я особи | санкційний + медіа скринінг |
| `1234567890 --person` | РНОКПП (10 цифр) | перевірка фізособи |
| `"ПриватБанк"` | назва компанії | резолвинг → кандидати або перевірка |

Резолвинг назви чистить організаційні форми (ТОВ/LLC/JSC), лапки, пробує
варіанти транслітерації (латиниця ↔ кирилиця). При неоднозначності повертає
кандидатів і просить уточнити ЄДРПОУ (не вгадує).

## 6. Приклад вихідного звіту

Приклад згенерованого звіту наведено в `examples/` (HTML, PDF, граф, JSON).

Консольний підсумок прогону для санкційної особи:

```
============================================================
  KYC ЗВІТ: Дерипаска Олег
============================================================
  Тип:         person
  Trust score: 32/100
  Рівень:      HIGH
  Покриття:    sanctions=checked, adverse_media=checked
  Артефакти:   json, html, pdf, graph, evidence_zip
============================================================
```

Санкційний скринінг для цього прогону знайшов збіги у п'яти списках:
РНБО, США (SDN), ЄС, Велика Британія, Канада.

Звіт містить:
- **trust score + рівень ризику** з причиною
- **покриття джерел** (checked / not_found / skipped / error)
- **red flags** — список тривожних сигналів
- **audit trail** — покроковий розклад балів із джерелами
- **дані суб'єкта** — реєстраційна інформація
- **санкційний скринінг** — збіги по списках із підставами й періодами
- **регуляторна історія** — стягнення НБУ/АМКУ (окремо від санкцій)
- **корпоративна структура** — пов'язані особи, граф зв'язків
- **evidence** — усі первинні дані в архіві

## 7. Джерела та API

| Сервіс | Призначення | Ключ (.env) |
|--------|-------------|-------------|
| **Opendatabot** | держреєстри України (компанії, ФОП, санкції РНБО/OFAC/ЄС/UK/Канада, PEP, регуляторні стягнення) | `OPENDATABOT_API_KEY` |
| **Serper** | пошук adverse media (Google SERP) | `SERPER_API_KEY` |
| **Anthropic Claude** | класифікація медіа-згадок | `ANTHROPIC_API_KEY` |

Санкційний скринінг повністю через Opendatabot — окремий ключ OpenSanctions
**не потрібен**. Ендпоінт `/person-sanctions` повертає міжнародні списки
(РНБО, США SDN/Non-SDN, ЄС, Велика Британія, Канада).

Використані ендпоінти Opendatabot: `/full-company/{code}`, `/fop/{code}`,
`/person-sanctions`, `/pep`, `/search/companies-by-field`.

**Свіжість даних:** Opendatabot проксіює офіційні держреєстри України — дані
актуальні на момент запиту.

## 8. Відомі обмеження

Система свідомо проектувалась із чесним відображенням своїх меж:

- **UA-центричність.** Покриття обмежене українськими реєстрами. Іноземний
  суб'єкт (Кіпр, Естонія тощо) у реєстрі **не знайдеться** — і це позначається
  як «не знайдено», а **не** як «чисто». Перехресна перевірка іноземних
  юрисдикцій (GLEIF, OpenCorporates) — у дорожній карті.
- **Тезки без ідентифікатора.** Перевірка фізособи за самим лише ПІБ (без
  РНОКПП/дати народження) ризикує хибною атрибуцією — однофамільців можна
  переплутати. Рятує INCONCLUSIVE: система не дає «чисто» на неоднозначних
  даних. За наявності коду/дати проблема знімається entity-resolution модулем.
- **Один реєстр — одна точка відмови.** Реєстрова частина тримається на
  Opendatabot; перехресної перевірки реєстрових фактів за другим незалежним
  джерелом немає.
- **PEP-перевірка особи потребує РНОКПП.** Worker фізосіб для пошуку лише за
  іменем (без коду) не виконує повний PEP-/афіляційний lookup; санкційний
  скринінг при цьому працює повноцінно.
- **Зв'язки особи (коло 1).** Opendatabot не надає «компанії конкретної особи»
  по РНОКПП. Обхід графа зв'язків — у дорожній карті.
- **Adverse media.** Якість залежить від пошукової видачі та класифікатора;
  можливі як хибнопозитивні, так і пропущені згадки. Класифікатор перевіряє
  релевантність згадки саме суб'єкту (відсіює однофамільців). Для компаній
  пошук іде за брендом (укр./англ.), а не за юр-назвою — це різко підвищує
  покриття; повноцінний пошук за англомовними ризик-запитами для міжнародних
  джерел (Kyiv Independent тощо) — у дорожній карті.
- **Межа автоматизації.** Система — помічник, не заміна аналітика. Рішення про
  відмову клієнту не можна повністю автоматизувати (регуляторна та етична
  вимога). Тому передбачено INCONCLUSIVE і явний дисклеймер у звіті.

## Архітектура

```
вхід (ЄДРПОУ / РНОКПП / ім'я / назва)
   │
   ├─ input_resolver ── визначення типу + резолвинг назви → ЄДРПОУ (асиметрично)
   │
   ├─ Worker 1 (registry_task / person_task) ── Opendatabot (реєстр)
   ├─ Worker 2 (sanctions_task) ────────────── Opendatabot (санкції, list-complete)
   ├─ Worker 3 (run_adverse_media) ─────────── Serper + Claude (медіа)
   │
   └─ aggregator ── entity resolution + trust scoring + audit trail
                    → JSON + HTML + PDF + граф + evidence ZIP
```

Усі джерела зшиваються під єдиним `slug` (напр. `company_14360570`,
`person_<рнокпп>`), що дозволяє агрегатору зібрати повну картину суб'єкта.
Оркестратор `run_kyc` працює синхронно (реєстр → санкції → медіа → звіт),
тож для компаній санкційний крок завжди читає вже зібраний реєстр.

### Дорожня карта
- **Phase 3 — мережевий аналіз та інтеграція:** пакетна перевірка (контрагент
  + пов'язані особи першого кола) одним зведеним звітом; обхід графа зв'язків з
  обмеженням глибини та бюджету викликів; SKILL для оркестрації з основного
  KYC-процесу; єдина git-бібліотека скілів.
- **Phase 4 — продуктизація:** зручний інтерфейс для аналітиків без терміналу;
  калібрування скорингу; версіонування перевірок (snapshot на дату рішення).
- **Phase 5 — масштаб:** іноземні юрисдикції (GLEIF, OpenCorporates); спільна
  база (SQLite) з дедуплікацією, TTL-кешем і виявленням мереж; кешування
  санкцій (list-complete = 6 запитів/особа).
