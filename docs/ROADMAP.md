# KYC Network Scout — Бачення та дорожня карта

> Жива стратегічна карта. Поточний стан (веб-міст + батч + скоринг ризику) —
> **тимчасовий зріз** ширшої цілі.

---

## 1. Бачення: внутрішній «Палантір» компанії

Це **не агрегатор реєстрів і не SaaS-дашборд**. Це **внутрішня
платформа-розслідування** (entity-intelligence engine), орієнтована на
**розкриття і поєднання даних** про особу, компанію та їхнє оточення.

| | YouControl / LIGA | KYC Network Scout |
|---|---|---|
| Суть | супермаркет структурованих даних | **рушій-розслідування** навколо сутності |
| Питання | «що записано в реєстрах?» | «**хто це, що це, чому це** — з усього відкритого простору» |
| Джерела | власні закриті БД | реєстри + санкції + медіа + OSINT + RF-сліди |
| Форма | дашборд для кінцевого користувача | **компонент, яким керує наш ШІ-скіл**, інтегрований у наш процес |
| Рів | широта даних | **трасабельність + чесне розрізнення доказ/лід** |

Внутрішній інструмент команди (один спільний інстанс), а не публічний продукт.
Конкурувати з YouControl по широті реєстрових даних — програшно; ніша —
**ширше й чесніше джерел, з наскрізною трасабельністю**.

---

## 2. Центральний об'єкт — сутність + граф (не звіт)

Зміщення фокусу: головний об'єкт системи — **сутність та її мережа зв'язків**,
а не PDF/HTML-звіт.

- **Entity (особа / компанія)** — ядро. Звіт — лише *один погляд* на сутність.
- **Розслідування — pivot-оване**: від вузла до пов'язаного вузла; граф росте.
  Не один постріл, а перехід «копнув головного → копнув його директора → …».
- **Скоринг ризику — одна лінза**, а не продукт. Бал — один із виходів поверх
  зібраної картини.

---

## 3. Принципи

**Розподіл праці:** **Scout = детермінований збирач доказів і факторів**
(1 суб'єкт → 1 структурований результат; не вирішує *кого* перевіряти і не
зводить картину). **KYC-скіл = оркестрація + судження** (розгортає граф,
інтерпретує, зводить, рахує фінальний вердикт).

**Три шари «детермінізму»:** процес/повнота — святі; LLM-інтерпретація медіа —
відтворювана через evidence+trace; диференціація джерел — детерміновані ПРАВИЛА.

**Провенанс наскрізь (Palantir-grade lineage):** для кожної крихти даних — звідки,
коли, з якою довірою. Будь-хто має простежити *що* шукали, *як*, *чому*.

---

## 4. Tier'и довіри джерел — ядро і рів

Те, що робить це **відповідальним** Палантіром, а не чорною скринькою. Кожне
джерело має рівень довіри, і він визначає, як факт використовується:

| Tier | Приклади | Як використовується |
|---|---|---|
| **Доказ** (high) | держреєстри, санкційні списки | впливає на бал; можна показати регулятору |
| **Сигнал** (medium) | відоме медіа, журналістські розслідування | впливає обережно; підкріплює/спростовує |
| **Лід** (low) | зливи, анонімні бази, форуми | **лише напрямок копати**; «перевір руками»; **ніколи не як доказ у балі** |

Так OSINT-ширина (зокрема RF-сліди, зливи) досягається **без** втрати
захищеності: зливи дають *куди копати*, а вердикт тримається на легітимному.
Це та сама філософія INCONCLUSIVE — але на рівні джерела.

---

## 5. Обмеження реєстрів (тверді стіни — задокументовано)

Перевірено на реальних даних і коді клієнта Opendatabot:

| Зв'язок | Доступність |
|---|---|
| Компанія → директор / бенефіціари | ✅ **лише ім'я** (без коду; 0 з 9 бенефіціарів мали код) |
| Компанія → адреса | ✅ (можливі співмешканці) |
| **Особа → її компанії** | ❌ стіна (`unavailable_via_api`) |
| Бенефіціар → його сутність (код) | ❌ коду нема → не резолвиться надійно |

Наслідок: надійний обхід — лише **компанія → її люди (name-only)**. Зворотний
обхід і резолвінг ідентичності — заблоковані поточним API. Іноземна юрисдикція
(OpenCorporates / GLEIF) — головний кандидат на розблокування ширшого графа.

---

## 6. Інтерфейс Scout ↔ скіл (цільовий контракт)

Scout на один виклик (одна сутність) повертає:

```
{ subject, factors/signals, related_candidates, reference_verdict,
  coverage, evidence, trace }   # кожен факт — з джерелом і tier'ом довіри
```

Скіл: оркеструє граф (pivot по related_candidates), бере reference_verdict
**або** перераховує за власними вагами, зводить картину сутності.

---

## 7. Фази (як цеглини entity-investigation платформи)

**Фаза 1 — Контракт Scout↔скіл + traceability/провенанс** (фундамент)
- Стабільний контракт (розділ 6); кожен факт несе джерело + tier довіри + trace.

**Фаза 2 — Граф зв'язків, коло 1** (ядро pivot-розслідування)
- Сутність → директор/бенефіціари → скринінг кожного. Чесно name-only /
  INCONCLUSIVE (обмеження реєстру). Ліміт глибини, бюджет, дедуплікація.
- Паралельно: розвідка джерел кодів (OpenCorporates) для ширшого графа.

**Фаза 3 — Іноземна юрисдикція + OSINT-розширення джерел**
- Хоча б одна іноз. юрисдикція як джерело зв'язків «особа ↔ компанії».
- Нові OSINT-джерела (медіа, RF-сліди) у той самий конвеєр під tier'ами довіри.

**Фаза 4 — Спільна БД як object-store сутностей + версіонування**
- SQLite: сутності + зв'язки, дедуплікація, TTL-кеш, виявлення мереж.
- **Snapshot на дату + піврічні переперевірки з diff**: «що змінилось у
  контрагента» — система пам'ятає сутність у часі.

**Паралельний трек — Аналітична якість** (тепер центральний, бо це рів):
- Tier'и довіри джерел (доказ/сигнал/лід) — розділ 4.
- Релевантність судових справ (роль/предмет/масштаб) — прибрати «банк-позивач = ризик».

---

## 8. Реалістична межа

Палантір цілком — це fusion + онтологія + access control + lineage + роки. Нам
**не треба все**. Ядро, що робить рамку реальною: **entity-граф** (об'єкт) +
**мульти-джерельна OSINT-фузія** + **провенанс/tier'и** + **pivot-розслідування**.
Рухаємось «помаленьку, але в правильному напрямку» — спершу чесна мінімальна
мережа, далі поступове розширення джерел і кола.
