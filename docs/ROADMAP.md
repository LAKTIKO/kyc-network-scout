# KYC Network Scout — Дорожня карта

> Жива дорожня карта інтеграції. Поточний стан (веб-міст + батч + скоринг
> ризику) — **тимчасовий місток**; стратегічна ціль — інтеграція Scout як
> детермінованого підмодуля в KYC-скіл (оркестратор на базі Claude).

---

## 1. Принципи архітектури

**Розподіл праці:**
- **Scout = детермінований збирач доказів і факторів.** Один суб'єкт на вхід →
  один структурований звіт на вихід. Scout **ніколи** не вирішує, *кого*
  перевіряти, і не зводить картину між суб'єктами.
- **KYC-скіл = шар судження й оркестрації.** Вирішує кого/як глибоко перевіряти,
  розгортає мережу зв'язків, інтерпретує, зводить підсумкову картину. Скіл
  **ніколи** не збирає сирі дані сам.

**Три шари «детермінізму»** (знімають напругу «детермінований, але аналізує
джерела інакше»):
1. **Процес і повнота — святі й детерміновані.** Завжди ті самі джерела, той
   самий перебір санкційних списків, та сама стратегія запитів. Це compliance-
   цінність: «як саме ви перевіряли?» → відповідь у коді.
2. **LLM-інтерпретація медіа — не бітово-детермінована, але відтворювана через
   evidence + trace.** Фіксуємо вхід (статтю), вихід (класифікацію), причину.
3. **Диференціація джерел — детерміновані ПРАВИЛА** (а не «настрій моделі»):
   tier'и джерел, «банк-позивач ≠ ризик» тощо.

**Traceability наскрізь.** Будь-хто, навіть «хто нічого не розуміє», має
простежити *що* шукали, *як*, *з яких джерел* і *чому* такий висновок.

## 2. Інтерфейс Scout ↔ скіл (цільовий контракт)

Scout на один виклик (один суб'єкт) повертає:

```
{
  subject,              # хто перевірявся
  factors / signals,    # сирі структуровані фактори (реєстр, санкції, медіа)
  related_candidates,   # кандидати-в-пов'язані (директор, бенефіціари, ...)
  reference_verdict,    # довідковий ризик 0–100 за вбудованими правилами
  coverage,             # що перевірено / не перевірено (чесно)
  evidence,             # первинні джерела на момент перевірки
  trace                 # запити, джерела, знахідки, впевненість — searchable
}
```

Скіл: оркеструє мережу, бере `reference_verdict` **або** перераховує за власними
вагами поверх `factors`, зводить картину. Гібрид: код дає детермінований збір +
довідковий вердикт; скіл лишає за собою фінальне судження.

## 3. Обмеження реєстрів (тверді стіни — задокументовано)

Перевірено на реальних даних і коді клієнта Opendatabot:

| Зв'язок | Доступність |
|---|---|
| Компанія → директор | ✅ **лише ім'я** (без коду) |
| Компанія → бенефіціари | ✅ **лише ім'я** (0 з 9 у вибірці мали код) |
| Компанія → адреса | ✅ (можливі співмешканці за полем) |
| **Особа → її компанії** | ❌ стіна (`unavailable_via_api`) |
| Бенефіціар → його сутність (код) | ❌ коду нема → не резолвиться надійно |

Наслідок: надійний обхід — лише **компанія → її люди (name-only)**. Зворотний
обхід (особа → компанії) і резолвінг ідентичності пов'язаних — заблоковані
поточним API. Це формує дизайн мережі (нижче).

---

## 4. Фази

### Фаза 1 — Інтерфейс Scout ↔ скіл + traceability (фундамент)
- Стабільний контракт виклику (розділ 2): структуровані факти + кандидати +
  reference-вердикт + evidence + trace.
- `trace`-об'єкт у форматі, який легко шукати («покажи, звідки цей сигнал»).
- Scout віддає факти; рахунок фінального вердикту може лишитися за скілом.

### Фаза 2 — Мережа зв'язків, коло 1 (мінімальна, чесно слабка)
- **Обсяг (за обмеженнями реєстру):** компанія → директор + бенефіціари →
  по кожному санкції + PEP + медіа **за іменем**.
- Майже кожен вузол буде name-only → часто **INCONCLUSIVE**. Це не баг:
  звіт показує «ось пов'язані особи; ось що *змогли* і *не змогли* підтвердити».
- Ліміт глибини (1), бюджет викликів, дедуплікація.
- **Паралельний розвідувальний трек:** чи дає OpenCorporates / GLEIF коди для
  пов'язаних осіб — це розблокування ширшої мережі (веде у Фазу 3).

### Фаза 3 — Іноземна юрисдикція + санкції юросіб (розблокування мережі)
- Хоча б одна іноземна юрисдикція (OpenCorporates / GLEIF) — зокрема як джерело
  зв'язків «особа ↔ компанії», яких Opendatabot не дає.
- Мультиюрисдикційний скринінг санкцій **і для компаній**.

### Фаза 4 — Спільна БД + версіонування
- SQLite (чи інше): дедуплікація, TTL-кеш, виявлення мереж, **snapshot на дату**.
- **Періодичні (піврічні) переперевірки з diff:** «що змінилось у контрагента
  від минулого разу».
- Частково підпирає Фазу 2 (кеш/дедуп здешевлюють обхід кола-1).

### Паралельний трек — Аналітична якість (вплітається між фазами)
- Релевантність судових справ (роль сторони / предмет / масштаб) — прибрати
  «банк-позивач = ризик».
- Градація якості джерел (доменні tier'и).

---

## 5. Порядок і залежності

```
Ф1 (інтерфейс) ──► Ф2 (мережа name-only) ──► Ф4 (БД, версіонування)
                        │
                        └─ паралельно: розвідка Ф3 (іноз. юрисдикція = коди)
                                            └──► Ф3 розблоковує ширшу мережу
Якість (суд. справи, tier'и джерел) — вплітається будь-коли.
```

Принцип: рухаємось «помаленьку, але в правильному напрямку» — спершу чесна
мінімальна мережа, далі поступове розблокування ширшого кола в міру появи
джерел ідентифікаторів.
